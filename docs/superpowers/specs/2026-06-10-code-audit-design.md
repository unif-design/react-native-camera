# 代码深度审查 —— 设计文档

日期：2026-06-10
状态：已与用户逐节确认

## 背景与目标

对 `@unif/react-native-camera`（vision-camera 5.x 封装的弹窗式相机库）做一次全量深度审查，从六个角度产出审查报告：**技术选型、代码架构、代码质量、代码重复率、官方 API / llms 最佳实践对齐、代码规范**。

## 范围

- **审**：根 `src/`（约 50 个源文件 + 41 个测试文件，约 6300 行）+ 工程配置层（`package.json`、`tsconfig*.json`、eslint、jest、builder-bob、lefthook、`.github/` CI）。
- **不审**：`example/`（宿主调试 app）与 `website/`（文档站）——仅在影响库结论时顺带查看。
- **边界**：只出报告，**不改任何代码**。修复由用户读完报告后另行决策。

## 交付物

`docs/audit/2026-06-10-code-audit.md` —— 仓内 Markdown 审查报告，提交进 git。

### 报告结构（已确认）

1. **执行摘要** —— 总体结论 + 六维度各一行评价（优 / 良 / 需改进）+ Top 发现速览表
2. **硬数据快照** —— LOC、测试数与覆盖率、jscpd 重复率、tsc / eslint 现状（全部为工具真实输出）
3. **分维度详查**（六章）—— 每条发现项固定格式：
   - 严重度：**P0**（缺陷，应尽快修）/ **P1**（明确问题，建议修）/ **P2**（改进机会）/ **P3**（风格、可选）
   - 位置 `file:line` · 问题描述 · 依据（官方文档 URL / 工具输出）· 修复建议（只给方向与理由，不展开实现）
4. **做对的地方** —— 经核实的良好实践亮点（确认哪些设计是对的、不该动）
5. **附录** —— 方法论、各镜头覆盖范围、静态分析局限声明

发现项**按严重度排序**，不按文件排序。

## 审查方法（方案 A，已确认）

### 第一层：工具硬数据（主会话执行）

| 工具 | 取什么 |
| --- | --- |
| `yarn typecheck` | tsc 现状是否干净 |
| `yarn lint` | eslint 现状 |
| `yarn test --coverage` | 测试通过率 + 覆盖率数字 |
| `npx jscpd src` | 重复率真实数字（排除 `__tests__` 一次、含测试对照一次） |
| 人工核对 | `package.json#exports` / peerDeps 与实际 import 的一致性 |

### 第二层：五个并行 subagent 镜头

每个 agent 独立通读源码，只带自己的视角：

| 镜头 | 关注点 |
| --- | --- |
| 架构 | 模块边界、5-hook 拆分合理性、数据流、导出面收敛度、组件耦合 |
| 质量 | 逻辑正确性、边界条件、错误处理、内存管理（Skia dispose / ref 守卫 / 竞态）、性能 |
| 重复与简化 | jscpd 结果人工复核 + 语义重复（工具测不出的同构逻辑）、可简化点 |
| 技术选型与最佳实践（联网） | ① 技术选型：peerDeps 选择合理性（如 @dr.pogodin/react-native-fs fork、nitro 栈、reanimated-carousel）、版本策略、RN 0.85 新架构适配；② API 对齐：核对 vision-camera 5.x / reanimated 4.x / Skia / RN 0.85 官方文档与本库 llms.txt——API 是否最新、有无弃用用法、是否符合官方推荐模式 |
| 规范与工程 | eslint / tsconfig 严格度、命名一致性、CLAUDE.md 中文 why 注释标准执行度、测试规范、builder-bob / CI / lefthook 配置 |

镜头与报告章节的对应：五镜头供稿六章——「技术选型与最佳实践」镜头同时供稿「技术选型」与「最佳实践对齐」两章，其余镜头一对一。

### 第三层：主会话核实与汇总（防幻觉硬门槛）

- agent 的发现 ≠ 报告内容——每条发现由主会话**读原文核实**文件、行号、真伪后才入报
- 官方文档结论必须附出处 URL；联网失败则标注「未能核对」而非断言
- 静态分析测不了的（真机相机行为、GPU 水印效果）明确标为局限
- 跨镜头重复的发现去重合并，严重度取核实后的判断

## 错误处理

- **工具跑失败**：jscpd 不可用时改用 grep 近似估计并在报告注明方法差异；coverage 跑不出则只报测试通过率
- **agent 返回空 / 失败**：重试一次，仍失败则该维度由主会话直接深读兜底
- **联网失败**：退回本仓 CLAUDE.md 与既有知识，对应结论标注「未能联网核对」

## 成功标准

- 六个维度每个都有结论（有发现列发现，无发现明确说「未发现问题」）
- 每条入报发现都有 `file:line` 且经主会话核实
- 重复率、覆盖率有真实数字
- 最佳实践章节的关键结论有官方文档出处
- 报告可直接作为后续修复任务的输入（发现项可逐个认领）

## 非目标

- 不修任何代码（含「顺手修」）
- 不审 example/ 与 website/ 的代码质量
- 不做真机行为验证（静态审查局限，报告中声明）
